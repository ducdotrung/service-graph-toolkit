PoC narrative

Faster delivery needs a control loop.

The demo is not three separate AI agents. It is an engineering knowledge platform: build trusted context from code, config, runtime signals, and human inventory; expose it through docs, graph Q&A, skills, and alert response; then measure whether AI-assisted delivery is faster and still safe.

Thesis

AI increases engineering output. The bottleneck moves from writing code to proving correctness, understanding impact, reviewing change, and responding to incidents. This platform targets that new bottleneck.

Demo assets

GitNexus graph + inventory, Code Atlas docs/Q&A, and alert-agent triage form one loop from code understanding to production feedback.

Market problem

The Acceleration Whiplash

The Faros AI Engineering Report 2026 describes a split outcome: business throughput improves, but downstream quality and review pressure rise at the same time. That is the problem this platform is designed to manage.

Root-cause bridge

AI is smart, but context-poor AI is fragile

Hallucination is not only a model problem. It is often a context supply-chain problem: the model is asked to reason about a codebase without the right service map, contracts, configs, owners, runtime signals, and team delivery rules.

1. More AI-generated work

AI makes it easy to create code, PRs, docs, product specs, and tests. Output accelerates before the surrounding review, impact analysis, and incident response systems are upgraded.

2. More uncertainty downstream

Reviewers and agents must infer hidden dependencies, runtime configuration, data contracts, and team conventions from scattered repositories. That is where hallucination and missed impact enter.

3. Better context fixes the root

Feed the AI correct, fresh, human-correctable context: code graph, inventory, configs, CI/CD, GitNexus indexes, PR history, docs, alerts, and eventually tracing and product specs.

The platform goal is to turn engineering context into a maintained product. AI agents become safer when they reason from that product instead of guessing from raw files.

Freshness

Easy to rebuild

The local workflow is rebuilt on demand. A shared-workstation template can validate, re-index, and regenerate declared project sources on a schedule, without automatic cloning or pulling; PR-triggered refresh remains a later option.

Human in the loop

Correct the graph, do not blindly trust it

AI can propose inventory updates, but humans can edit `inventory.yaml` to fix ownership, service links, routes, and edge confidence. That correction then feeds every AI surface.

Inventory sample

What humans actually edit

The inventory is deliberately boring YAML. That is the point: AI can propose changes, but engineers can review and correct service ownership, repo paths, deployment bindings, and cross-service edges with evidence.

# inventory.yaml is the source of truth for cross-service facts. # Every important service or edge should cite evidence; otherwise it is a guess. repos: backend-core: path: backend-core type: monorepo primary_lang: java build: maven chart_path: chart services: bio-service: repo: backend-core role: backend framework: spring-boot paths: app_module: backend-core/bio-service spring_resources: backend-core/bio-service/src/main/resources chart: kind: local path: backend-core/chart edges: - from: gateway-service to: bio-service kind: gateway_route evidence: backend-core/gateway-service/src/main/resources/application.yml - from: bio-service to: identity-service kind: feign_call evidence: backend-core/bio-service/**/client/Identity*.java
Why YAML

Readable by humans

The artifact is simple enough for engineers to review in PRs and correct when AI extraction is wrong.

Why evidence

No source, no trust

Every edge should point back to code, config, deployment files, route definitions, traces, or product-spec evidence. That keeps agents from treating guesses as facts.

Why derived views

One truth, many surfaces

The YAML regenerates JSON for agents, service pages for humans, Mermaid maps for review, and group config for GitNexus.

Code intelligence layer

GitNexus turns many repos into one queryable platform graph

Each repo keeps its own `.gitnexus` index/ladybug database for local code intelligence. The group database connects those indexes through the inventory bridge, so AI can reason across the shared code platform instead of one repo at a time.

Operational thesis

From noise to action

  • Sentry is currently the active source, but the runtime has a source-plugin shape for future alert sources.
  • The flow is Source -> Triage -> Review -> Recommendation -> Sender -> Teams.
  • Human review decisions can feed self-improvement rules instead of retraining from scratch.
  • The KPI to prove next is 30-day filter rate plus accepted recommendation rate.
  • Current dashboard math: 768 classified alerts, 7 final send decisions, so only 0.91% are real sends and 99.09% are filtered, held, or not sent.
  • As incidents rise, the alert layer becomes a budget-efficient way to keep response time low without adding a full-time manual triage team.
  • Error count, incident count, ignored-alert ratio, and false-negative review become the safety metrics that can slow or redirect AI acceleration.

Quality brake

Measure when to pull back

The Faros warning is that output metrics can look healthy while incidents and bugs rise. Alert metrics give engineering leadership a counter-signal: if error rate, incident rate, ignored alert rate, or rollback reasons spike, slow the workflow and invest in tests, review, or platform context.

What's next

Three tools, three next tracks

The PoC proves the loop. The next phase should make each part production-useful: richer knowledge base, stronger docs surface, and broader alert/control signals.

Big next point

Strategic alerts, not only ops alerts

The bigger alert opportunity is not another pager feed. It is a delivery-risk layer: combine issue tracker, incidents, rollbacks, hotfixes, review backlog, escaped defects, and repeated alert classes into pull-back warnings for leadership when AI throughput rises faster than quality control.

Big next point

Two missing truth planes

Tracing adds observed runtime truth. product specs add planned product intent. Together with code and deploy config, they let the platform compare what the system should do, what the code says, and what production actually does.

1. Knowledge base

Add truth signals and skills

  • Add tracing as observed runtime truth: which services actually call each other, which routes are hot, and which paths appear in real traffic.
  • Keep tracing limitations explicit: it only sees executed paths, so rare edge cases and new features may be missing until traffic reaches them.
  • Add product specs and product specs as planned-intent evidence, then compare planned behavior against code, routes, tests, and runtime traces.
  • Promote completed PRs and release branches into replay/eval datasets so each real implementation improves future skills.
  • Add more useful skills: implementation planning, PR impact review, QA test-case generation, regression-risk analysis, product spec review, release-risk summary, and PM/BA estimation support.
  • Feed this knowledge base into existing QA/dev skills instead of building isolated agents with their own stale context.
2. Wiki / Code Atlas

Make graph-aware docs real

  • Finish platform mode as a first-class mode, not just a repo knowledge docs with extra context.
  • Use deterministic page planning so the docs tree is stable before LLM generation starts.
  • Render evidence panels for architecture claims: source file, inventory edge, GitNexus result, route config, chart/deploy context.
  • Make chat query routing explicit: repo-local code question, cross-service impact question, route/event-flow question, owner/evidence question.
  • Integrate live GitNexus/MCP lookup for on-demand code-level details instead of only static graph artifacts.
  • Improve cache invalidation and refresh UX: show graph freshness, source commit, index age, and missing-index warnings.
  • Add export/sharing: HTML/PDF handout, service dossier, impact report, and product spec/QA review report.
  • Add tests for repo mode, platform adapter, page planner, evidence rendering, and graph-aware chat.
3. Alert agents

Broaden sources and add pull-back warnings

  • Add more sources through the existing source-plugin design: Grafana, Azure Monitor, Kubernetes events, Prometheus/Alertmanager, gateway logs, CI/CD failures, and deployment events.
  • Keep Sentry as the current app-error source, but join alerts to service graph nodes, owners, routes, recent PRs, deployments, and downstream consumers.
  • Add Ops alerts, not only app exceptions: pod restarts, OOMKilled, HPA saturation, queue lag, database dependency errors, ingress/gateway error spikes, and cloud resource failures.
  • Add issue tracker and incident metrics as a second, higher-level alert layer: incident count, incident-to-PR ratio, rollback rate, hotfix rate, stale review queue, escaped defects, and repeated alert classes.
  • Build pull-back warnings: if incidents, rollbacks, P0/P1 alerts, MTTR, or ignored-alert rate rise while PR/AI throughput rises, flag leadership to slow delivery or invest in tests/review/context.
  • Use the current 99.09% not-sent rate as the noise-filter baseline, then measure accepted sends, false negatives, and response-time improvement.
